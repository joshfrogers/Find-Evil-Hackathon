# Accuracy Report

> **SANS "Find Evil!" submission requirement #6.** This document is deliberately self-critical. Agentic SIFT is an autonomous forensic-analysis CLI (a Python orchestrator that drives SANS SIFT tools and uses Claude for planning and interpretation). We believe an honest accuracy report — one that names our failures, quantifies them against ground truth, and shows the hallucinations our own pipeline catches — is more useful to a working analyst than a polished one. Every number below is reproducible from the scorer in `accuracy/` and the audit logs each run emits. Where a claim is an engineering judgment rather than a measured fact, we say so.

---

## 1. Methodology

### 1.1 Ground-truth baselines

Accuracy is measured by comparing each generated report against a hand-built **ground-truth baseline** (an "answer key") for a specific evidence image. A baseline is a JSON file (`accuracy/baseline.py`, schema in `tests/fixtures/baselines/`) listing every artifact a correct investigation should surface. Each baseline finding carries:

- `id`, `description`
- optional `ioc_type` / `ioc_value` (e.g. `file_path`, `registry_key`, `hash`, `ip`, `email`)
- `must_find` — whether a miss counts against recall. `must_find=True` items form the **recall denominator**; `must_find=False` items are bonus credit (matching them helps, missing them does not hurt).

The public datasets we score against are open: the **NIST CFReDS "Hacking Case"** image (the "Greg Schardt / Mr. Evil" laptop) is our primary benchmark, along with NIST data-leak and other public images.

> **Honest caveat on the baselines themselves.** Our baselines are not yet uniformly trustworthy. Some early baselines were transcribed from public blog write-ups and not independently verified against the image, and one fixture was synthetic (placeholder IOCs) used only for plumbing tests. Building and validating **one fully analyst-verified golden report per benchmark image** — confirming every `must_find` artifact actually exists on the disk — is in progress. Treat precision/recall figures as accurate *relative to the current baseline*, which is itself being hardened.

### 1.2 Matching: how a finding "counts"

`accuracy/scorer.py` matches each agent finding to **at most one** baseline item, greedy first-match-wins, in priority order (highest precision first):

1. **Exact normalized IOC match** — IOC values are normalized (lowercasing, slash normalization) using the same helper that powers cross-domain corroboration, so scoring and corroboration agree. IOC types must be compatible (identical, or one side blank).
2. **Path-segment IOC match** — for `file_path` / `registry_key` IOCs, a finding matches when one path's segments appear as a contiguous run (≥2 segments) inside the other's. This aligns a mount-rooted agent path (`/.../mnt/vol0/Program Files/mIRC/mirc.ini`) with a baseline's Windows-absolute path (`C:\Program Files\mIRC\mirc.ini`), and a registry key+value with a key-only baseline IOC. The ≥2-segment floor avoids matching on a single common leaf like `config`.
3. **Description fuzzy match** — `difflib.SequenceMatcher` similarity above a 0.55 threshold, for paraphrased findings with no IOC (e.g. timestomping, prefetch-execution proof).

A baseline item, once matched, cannot match a second finding — so duplicate agent findings about the same artifact do **not** inflate recall (but they *do* hurt precision; see §3).

### 1.3 Metrics

Standard definitions (`_compute_metrics`):

- **Precision** = TP / (TP + FP) — fraction of agent findings that matched a baseline item.
- **Recall** = TP / required_baseline_findings — fraction of `must_find` items the agent surfaced.
- **F1** = harmonic mean of precision and recall.
- **Hallucination rate (uncaught)** = flagged hallucinations / total agent findings (see §4).

### 1.4 Confidence tiers (confirmed vs inferred — Criterion 2)

Every finding carries a confidence tier, assigned by the agent and re-checked by the verifier:

| Tier | Meaning |
|------|---------|
| **Confirmed** | Direct evidence in tool output (e.g. a registry value read straight from a hive). |
| **Inferred** | Correlated / indirect — supported by reasoning across artifacts, not a single direct read. |
| **Possible** | A single weak signal; flagged for human follow-up. |

The scorer reports a `confirmed_vs_inferred` breakdown so a reviewer can immediately see how much of a report is direct evidence versus inference. This is the explicit "distinguish confirmed from inferred" ask in Criterion 2, and we treat it as a first-class output, not a footnote.

## 3. Honest Weaknesses

### 3.1 The flat-zero runs are real and were our biggest early problem

More than half the rows above are `0.000` across every metric. These were **not** investigations that ran cleanly and missed everything — they were broken runs:

- A **transport regression** (an unconditionally-sent prompt-caching beta header rejected by the LLM gateway with HTTP 400) would have broken *every* LLM call until caught and fixed.
- A **formatting crash** in tool-list rendering aborted dispatch and produced 0 findings.
- Early tool-discovery handed agents an oversized, unscoped tool menu, causing enormous, slow runs that timed out before converging.

We are reporting these because they are the honest history of the project, and because they illustrate a real risk of agentic forensics: **a pipeline failure can masquerade as "no findings."** Our mitigation is run status and phase markers in the audit log so a `0.000` run is distinguishable from a clean empty result — but a reviewer should always check run status before trusting a low score.

### 3.2 Low precision — root cause and fix in progress

This is the weakness we are least proud of and most focused on. In the well-characterized 3-round NIST run, the agent emitted **60 findings against 11 required baseline items**, yielding precision **0.100**. The recall (0.545) was respectable; the precision was poor. Root causes, all observed in real output:

1. **Redundant / near-duplicate findings.** The same artifact is reported multiple times by different domain agents and across rounds. For example, the EnCase acquisition metadata ("case Greg Schardt, examiner Shane Robinson, Dell Latitude CPi, MD5 `aee4fcd9…`") appears as Finding 3, Finding 4, Finding 23, Finding 29, and Finding 42 — five findings for one baseline fact. Because the scorer matches each baseline item only once, four of those five are counted as false positives. They are *true statements*, but they are not *distinct findings*, and they correctly drag precision down.

2. **Tool failures and tool-capability notes narrated as findings.** In an earlier reviewed run, roughly a third of "findings" were not forensic conclusions at all but execution artifacts — e.g. *"fls output was truncated"* (Finding 28 in the run above is exactly this), *"fls failed at offset 128," "strings returned noise," "ClamAV signature DB is stale."* These belong in a **Limitations** section, not the findings list. Each one is a false positive against the baseline.

3. **Granular sub-claims of one logical finding.** "Hacking toolkit installed" is split into one finding per tool (Cain & Abel, Ethereal, WinPcap, Network Stumbler, Look@LAN, 123-Write-All-Passwords, Anonymizer, mIRC, CuteFTP…), each as a separate finding. Forensically reasonable; arithmetically punishing to precision when the baseline encodes "toolkit present" more coarsely.

**Fix in progress (designed, partially implemented):**
- **Evidence-positive guard at finding creation** — a finding must cite ≥1 *successful* evidence link (exit 0, not rejected); otherwise it is recorded as a `failed_approach`, never a finding.
- **Route tool-failure and tool-capability notes to a Limitations section**, never the findings list (sub-agent prompt + post-filter).
- **De-duplication / finding-merge** before report emission, so one artifact yields one finding.
- **Partial/killed runs mark findings as triage-only/unverified** rather than presenting them at parity with verified findings.

We expect these to raise precision substantially without materially hurting recall, because the items being removed are duplicates and non-findings, not true positives. That claim is an engineering expectation, **not yet a measured result** — it will be re-scored once the guard lands.

### 3.3 Specific missed findings (false negatives)

In the best (recall-0.727) run, the three missed `must_find` items were:

- **B-005 — host IP / MAC address (`192.168.1.111`).** This is a **persistent, content-level gap**, not a speed artifact. The network identity lives in a registry interface key that the RegRipper `nic2` plugin should emit, but our run did not surface the host's own IP/MAC. We consider this the one true content miss in the best run, and it is an explicit, tracked follow-up — it would not be fixed merely by running more rounds.
- **B-010 — Look@LAN install path.** The tool's *presence* was found and reported (it appears in the run), but the specific baseline-required install path was not surfaced as a matched finding. We assess this as **likely recoverable with additional rounds** — the best run did only one round.
- **B-011 — the address `mrevilrulez@yahoo.com` in application configuration.** This string lives inside an installed application's config file and requires reading that file's contents (not just enumerating the filesystem). Also assessed as **likely recoverable with deeper, multi-round file-content inspection.**

The distinction matters: **B-005 is a confirmed capability gap; B-010 and B-011 are inferred-recoverable** (a hypothesis about why they were missed, to be validated by a deeper run). We label them that way deliberately rather than claiming they are all "almost fixed."

There is a real **recall-vs-rounds tension**: the best *recall* run was the fast, single-round configuration; deeper runs found more nuanced artifacts but at the cost of even worse precision (more duplicates) and longer wall-clock. Tuning this tradeoff is ongoing.

## 4. Hallucinations

### 4.1 What the detector does

`accuracy/hallucination.py` cross-checks every finding against the structured audit log (`audit.jsonl`). The core rule:

> A finding is a **hallucination** if its `evidence_links` cannot be resolved to a *successful* `tool_execution` event in the audit log.

"Successful" is strict: the linked execution must exist in the audit log, have `exit_code == 0`, and not have been rejected by the executor. Three failure modes are flagged:

1. **No evidence links at all** — the finding has nothing backing it.
2. **Links not in the audit log** — the cited execution ID does not exist (a fabricated reference).
3. **Links to failed/rejected executions** — the cited tool ran but errored or was blocked, so it produced no output the model could legitimately interpret. Citing it is treated as a hallucination even though an event with that ID exists.

The detector also surfaces a **credit** bucket — hallucinations the system **caught itself**: findings the verifier returned a `refuted` verdict on, and `self_correction` events that downgraded confidence (e.g. confirmed → inferred). Under SANS Criterion 2, hallucinations the agent catches and documents count *for* us, not against us.

### 4.2 Measured rate: 0.0 — and why that number deserves scrutiny

Across all scored runs, the **measured uncaught-hallucination rate is 0.000.** In the characterized 3-round NIST run, the scorer recorded *"None — every finding traces back to a real tool execution,"* while separately crediting **16 hallucinations the system caught itself** (4 verifier refutations + 12 self-correction downgrades).

**We do not present 0.0 as proof the system cannot hallucinate.** It reflects a specific, enforced discipline:

- Findings are required to carry evidence links, and the report generator binds each finding to the exact command and execution ID that produced it (visible per-finding in every report).
- The adversarial verifier challenges high-value findings, removing or downgrading unsupported ones *before* they reach the final report (those 16 catches).
- The detector measures **traceability to a successful tool execution — not factual correctness of the interpretation.** A finding can be perfectly grounded in a real `fls` execution and still draw a *wrong forensic conclusion* from that output. That class of error is **not** captured by the hallucination rate; it shows up (if the baseline encodes it) as a precision/recall miss instead.

So the honest reading is: **0.0 means "we found no claims fabricated out of thin air," not "every conclusion is correct."** The verifier-plus-evidence-link discipline is what produces the 0.0, and that discipline is exactly what could be circumvented by a subtly-wrong-but-grounded interpretation. We name this gap explicitly.

## 5. Evidence Integrity

Evidence integrity is enforced **architecturally, in code — not via prompt instructions** (which an LLM could ignore). The protections:

- **Read-only mounting.** Disk images are attached read-only at the OS level (loopback with the read-only flag, then a read-only mount). The mounted filesystem is what sub-agents read from; the original image is never written.
- **Allowlist of read-only tools only.** The executor validates every proposed command against an allowlist; write/modify tools are not exposed. The LLM never calls `subprocess`/SSH directly — all execution passes through the executor's validation pipeline (tool allowlist → argument validation → path validation with symlink-escape detection → argv array, never `shell=True` → resource-limited execution → audited capture).
- **Path sandboxing.** Evidence paths must resolve under configured evidence roots; symlink-escape attempts are rejected and logged.
- **Output isolation.** All tool output, extracted artifacts, reports, and logs are written to a separate working directory — never into the evidence path.
- **SHA-256 integrity bracketing.** The evidence image is hashed **before** the investigation and **re-hashed on close**. An **integrity (spoliation) assertion** fires on session teardown: if the post-investigation hash does not equal the pre-investigation hash, the run is flagged as a spoliation failure. Teardown is crash-safe so the bracket and unmount run even if the investigation aborts.

This means chain-of-custody is preserved by construction: the agent *cannot* write to evidence because no write path is exposed to it, and the before/after hash bracket provides cryptographic proof that the bytes did not change.

## 6. Spoliation Testing

Per SANS requirement #6, we treat spoliation as something to **actively test**, and we treat any failure found as signal to document, not hide. The approach:

1. **Hash-bracket every run.** As above, every investigation SHA-256-hashes the image before and after and asserts equality on close. This is not a one-off test — it runs on *every* investigation, so each report is itself a spoliation test result.
2. **Adversarial prompting.** Deliberately instruct the agent to modify evidence, delete files, or remount the image writable, and confirm the architectural guardrails (allowlist + read-only mount + path sandbox) refuse the action and log the rejection — i.e. that the constraints hold even when the LLM is *told* to violate them. This is exactly where architectural enforcement beats prompt-based enforcement: the refusal does not depend on the model's cooperation.
3. **Document every failure mode found.** Any case where a guardrail is bypassed, or the post-run hash differs from the pre-run hash, is recorded as a failure mode rather than suppressed.

> **Honest status.** The hash-bracket and the read-only/allowlist architecture are implemented and run on every investigation. Systematic, scripted **adversarial spoliation prompting across a fixed attack-prompt suite** is part of our test plan and is being formalized as a repeatable harness; we report it here as designed-and-partially-exercised, not as a completed exhaustive audit. Across the scored runs in §2, no spoliation assertion failure was observed.

## 7. Summary of What We Are Claiming vs. Not Claiming

| We claim (measured / confirmed) | We do **not** claim (inferred / unproven) |
|---|---|
| Best run: recall **0.727 (8/11)** on the public NIST Hacking Case, 0.0 measured hallucination rate. | That the system finds everything — 3 required artifacts were missed, one a confirmed capability gap (B-005). |
| Every reported finding traces to a real, successful tool execution (0.0 uncaught-hallucination rate). | That every grounded conclusion is forensically *correct* — the detector measures traceability, not interpretation quality. |
| Evidence integrity is enforced in code (read-only mount, allowlist, SHA-256 bracket, spoliation assertion). | That adversarial spoliation testing is exhaustively complete — the scripted attack-prompt suite is still being formalized. |
| The system catches and downgrades its own bad findings (16 caught in one run). | That precision is acceptable — it is our weakest metric (best F1 ≈ 0.17) and the active focus of the in-progress fixes in §3.2. |

We would rather a judge or an analyst know precisely where this tool is weak than be surprised by it on real evidence. The precision problem is real, the missed IP/MAC is real, and the 0.0 hallucination rate is a product of enforced discipline rather than a guarantee. Those are the facts the numbers support.
